# Active AI Conversations — Design Document

- **Creation Date**: 2026-05-15
- **Branch**: `feature/active-conversations`
- **Author**: Flyer Li
- **Status**: Draft, pending user review

## 1. Background and Objectives

The Branch Panel in the VS Code extension currently has three sections: Plans & Notes, Changes, Memories.
This design adds a fourth section **Active AI Conversations**, inserted above Plans & Notes,
allowing users to see recent active AI conversations (Claude, Cursor, Codex, Gemini, OpenCode, Copilot CLI, Copilot Chat) directly in the current project,
and open a standalone Conversation Details Panel by clicking a conversation entry to view its complete transcript.

### User Value
- Unified entry point for conversations across multiple AI clients: users don't need to open each tool separately to review what they recently did.
- Decoupled from commits: no need to wait for a conversation to become a summary to review it; active ongoing conversations are also accessible.

### Non-Goals
- Do not sync transcripts from external tools in real-time (read from disk on-demand is sufficient).
- Do not implement branch-level filtering in this phase (the sessions data model currently lacks a branch field; forced mapping would introduce new storage issues).
- Do not reorganize the rendering logic inside Plans & Notes (only insert a new section before it).

## 2. User Experience

### 2.1 Section Position and Appearance
- Rendering location: in `renderBranch()` at [`SidebarScriptBuilder.ts:1967`](../../../vscode/src/views/SidebarScriptBuilder.ts),
  positioned **before** the `plans` section, reusing the existing generic `renderSection()` container framework (collapse header / body / bottom CTA).
- UI title: `CONVERSATIONS` (all caps, following the mockup screenshot; aligned with the visual style of existing `PLANS & NOTES` / `CHANGES` / `MEMORIES`). Note: "Active AI Conversations" mentioned in this document is only a design codename, not the UI display text.
- Collapse state defaults to expanded; state persistence follows the existing section collapse mechanism.
- **No buttons in the section-level toolbar** — no `+`, no `Refresh`, no `↗`, etc. Simplification rationale: conversations are generated by AI tools themselves (not created within the panel); refresh is handled by the existing `Refresh` button in the Branch tab top toolbar (the `iconButton('refresh', 'Refresh', 'refresh')` already registered at [`SidebarScriptBuilder.ts:391`](../../../vscode/src/views/SidebarScriptBuilder.ts)).
- Refresh flow: user clicks the Refresh button in the top toolbar → webview already sends `{ type: 'refresh' }` outbound message (at [`SidebarScriptBuilder.ts:406`](../../../vscode/src/views/SidebarScriptBuilder.ts)) → extension refresh handler fan-outs a `listActiveConversations()` call and pushes `branch:conversationsData` (see §7). Title is synced and re-read each time, with no caching.

### 2.2 Single Row Rendering
Each row from left to right:
1. **Icon** (chat bubble SVG, uniform coloring, following the icon size/padding rhythm of Plans & Notes).
2. **Title** (taken from the native title field of that source; fallback to truncated first user message when native title is unavailable. See §5).
3. **Provider Badge**: `Claude / Cursor / Codex / Gemini / OpenCode / Copilot / Copilot Chat`, colored according to existing `transcript-source-*` CSS classes (already used for source badges in summary details); missing colors are supplemented by SidebarCssBuilder.
4. **Message Count** (number, gray caption).
5. **Relative Time** (e.g., `2h ago` / `1d ago`, reusing existing relative-time formatter — find the ready-made one in SummaryScriptBuilder, no need to rewrite).

Row height, padding, and hover behavior follow the rhythm of Plans & Notes rows (reference user preference: UI consistency = rhythm alignment rather than pixel min-height).

### 2.3 Click Behavior
Clicking a row triggers a webview → extension `branch:openConversation` message,
and the extension creates/activates a standalone webview panel **Conversation Details Panel** (see §6).

### 2.4 Empty State
If there are no active sessions within the 2-day window for the current project, the section still displays with an empty state message in the body:

> `No active AI conversations in the last 2 days.`

### 2.5 Foreign Mode (viewing non-workspace repo / branch view)
Completely hide the entire section. Use the existing `isViewingForeign()` logic (at [`SidebarScriptBuilder.ts:928`](../../../vscode/src/views/SidebarScriptBuilder.ts)).
Rationale: active session data comes from the current workspace's local files; it has no meaning in cross-repo views.

## 3. Definition of "Active"

- **Time Window**: last **2 days** (48h), determined by each session's `updatedAt` field (ISO 8601).
- **Quantity Limit**: **no limit**, sorted by `updatedAt` in descending order.
- **Scope**: current workspace's `cwd`, all 7 sources included.
- **Sorting**: single dimension `updatedAt` descending; same time uses `sessionId` lexicographic order for stable sorting.

## 4. Data Model

### 4.1 New Aggregation Layer `ActiveSessionAggregator`

File: [`cli/src/core/ActiveSessionAggregator.ts`](../../../cli/src/core/ActiveSessionAggregator.ts) (new)

```typescript
import type { TranscriptSource } from "../Types.js";

export interface ActiveConversationItem {
	readonly sessionId: string;
	readonly source: TranscriptSource;       // "claude" | "cursor" | "codex" | ...
	readonly title: string;                  // source: native title field, fallback to truncated first user message, see §5
	readonly messageCount: number;           // transcript count, 0 if unreadable
	readonly updatedAt: string;              // ISO 8601
	readonly transcriptPath: string;         // used by ConversationDetailsPanel on click
}

export interface ListActiveOptions {
	readonly cwd: string;
	readonly windowMs: number;     // default 48h = 2 * 24 * 3600 * 1000
}

export async function listActiveConversations(opts: ListActiveOptions): Promise<readonly ActiveConversationItem[]>;
```

Implementation notes:
- Internally call 7 discoverers concurrently (lazy import + `Promise.allSettled`); failure of a single source does not affect others.
- Platform differences: path comparison uses `path.normalize` + Windows case-insensitive (following the user-recorded `feedback_windows_path_separator_and_case` rule).
- `messageCount` is calculated by streaming the transcript header lines; 0 if unreadable (fail-open).
- `title` behavior see §5.

### 4.2 Reuse Existing Discoverers
- Claude / Gemini / Codex: read `SessionInfo[]` from `<projectDir>/.jolli/jollimemory/sessions.json`, filter by `updatedAt`.
- Cursor / OpenCode / Copilot CLI / Copilot Chat: on-demand scan their respective local paths.
- These functions already exist ([`cli/src/core/CursorSessionDiscoverer.ts`](../../../cli/src/core/CursorSessionDiscoverer.ts), `Copilot*Discoverer.ts`, etc.); this phase does not modify their interfaces, only calls them in the aggregator.

## 5. Title Resolution Strategy (Native First, No LLM)

Each session's title is obtained strictly following the "native first → fallback truncate" flow, **without calling LLM at all** — since 4 major sources already have AI-generated / user-edited title fields, we simply mirror them without reinventing the wheel.

### 5.1 Priority
```
title = nativeTitleOf(session) ?? fallbackTruncate(transcript) ?? "(untitled session)"
```

`listActiveConversations` returns the final title synchronously, **with no intermediate caching layer, no sticky strategy, no async push updates**. Each scan re-reads the native field, naturally always up-to-date.

### 5.2 Native Title Sources for Each Source

| Source | Has Native Title | Field / Value Path | Existing Reader Status |
|---|---|---|---|
| **Claude** | ✅ | `type: "ai-title"` line in `<transcriptFile>.jsonl` with `aiTitle` field; Claude Code **continuously appends new lines**, take the **last** ai-title line value | Existing reader does not read this; needs new addition |
| **OpenCode** | ✅ | sqlite `session.title` column | [`OpenCodeSessionDiscoverer.ts:168`](../../../cli/src/core/OpenCodeSessionDiscoverer.ts) already SELECTs it, but does not expose it; need to add field to SessionInfo return |
| **Cursor** | ✅ | sqlite globalStorage `cursorDiskKV` table `composerData:<id>` JSON value `name` field (user-editable, may be empty) | Existing reader does not read this; needs new addition |
| **Copilot CLI** | ✅ | sqlite `sessions.summary` column (auto-generated session summary by product) | [`CopilotSessionDiscoverer.ts:61`](../../../cli/src/core/CopilotSessionDiscoverer.ts) does not SELECT this column; needs new addition |
| **Codex** | ❌ | `session_meta` only has id/cwd/timestamp, no title | — |
| **Gemini** | ❌ | session JSON only has sessionId/projectHash/startTime/messages, no title | — |
| **Copilot Chat** | ❌ | JSONL patch document structure (version/creationDate/requests[]), no title field | — |

New module [`cli/src/core/SessionTitleResolver.ts`](../../../cli/src/core/SessionTitleResolver.ts) centralizes native title resolution logic for each source (one small function per source), decoupled from the aggregator — this way, if a source adds/changes a title field in the future, it won't affect the aggregator.

### 5.3 Fallback Truncation Rule

When native field does not exist, is empty, or read fails:
- Stream-read the transcript, find the **first user message** text content (parse by each source's transcript schema, not generalized "first line is user").
- Take the first 60 Unicode code points (without breaking surrogate pairs), remove newlines / excess whitespace.
- If transcript is corrupted or has no user message → `"(untitled session)"`.

Reading uses streaming (`fs.createReadStream` + line splitter, close stream immediately after getting the first user message), avoiding full-text loading of large transcripts.

### 5.4 Failure Handling
- Native field read exception (corrupted file / DB locked / field empty) → automatically fall back to truncation.
- Fallback also fails → `"(untitled session)"`.
- Do not throw errors out of the aggregator; a single session's title read failure does not affect other sessions.

### 5.5 Performance
- sqlite SELECT single column: nanosecond-level.
- Claude transcript find last `ai-title`: implementation can forward-stream read entire file and record last hit (JSONL within a few MB is completely acceptable), or optimize with reverse chunk reading (no need in MVP).
- Fallback truncation: stream-read until first user message then close.
- **Entire `listActiveConversations` is synchronously blocking throughout, no background tasks**.

### 6.4 Multi-Panel Behavior
- Repeated click on the same `sessionId`: reveal existing panel (panel cached by sessionId), do not open new instance.
- Different sessionId: each opens its own panel (users can compare multiple conversations side-by-side).

New outbound (webview → extension):
```typescript

```

**No new `branch:reloadConversations`** — refresh flow completely reuses the existing `{ type: 'refresh' }` message from the Branch tab top toolbar:
- Entry point: `iconButton('refresh', 'Refresh', 'refresh')` at [`SidebarScriptBuilder.ts:391`](../../../vscode/src/views/SidebarScriptBuilder.ts).
- Webview send code: `if (action === 'refresh') postMessage({ type: 'refresh' })` at [`SidebarScriptBuilder.ts:406-408`](../../../vscode/src/views/SidebarScriptBuilder.ts).
- Extension refresh handler already has fan-out pattern (push plans/files/...), this phase adds one line there: call `listActiveConversations({ cwd, windowMs: 48h })` then push `branch:conversationsData`.
- Initial webview open also follows the init push flow (following `pushPlans()` pattern, §2.1 already stated).

No `branch:conversationTitleUpdated` async title push message — title is obtained synchronously, arriving together with `branch:conversationsData`.

`SidebarWebviewProvider` adds dependency: `activeSessionsProvider`, calling `pushConversations()` in the initial push flow (following `pushPlans()` pattern).

Lazy channel consistency (per user preference [[feedback_lazy_cache_key_consistency]]): cache keys at trigger / response-match / render-lookup must all use **`sessionId`** (string equality), all three places must be literally identical.

## 9. Performance and Concurrency

- **Cold Start**: when webview opens, extension synchronously runs `listActiveConversations()` once then pushes `branch:conversationsData`. Title is already in the result, no subsequent async supplementation.
- **Concurrency**: aggregator internally calls 7 sources with `Promise.allSettled` concurrently. No LLM calls, no queue, no rate limiting.
- **Disk IO**:
  - sqlite SELECT: extremely light, single connection.
  - Claude transcript get ai-title: forward-stream read entire file, record last hit line (MVP stage JSONL within a few MB is acceptable; if real-world testing shows large file pressure, optimize with reverse chunk reading later).
  - Fallback truncation: stream-read until first user message then close immediately.
- **No background tasks, no timers**: completely idle when user is not operating.

## 10. Security and Privacy

- **Do not send transcript content externally**. Entire title resolution completes locally (sqlite SELECT / file read), no LLM calls, no remote services. This is the biggest privacy simplification compared to the previous version (LLM summary).
- Exception paths do not log transcript content.
- sqlite reads use read-only mode (`mode: "readonly"`), impossible to accidentally modify external tool databases.

## 11. Cross-Package / Cross-Language Constraints

- `SidebarMessages` is a TypeScript definition, used only on the VS Code side. IntelliJ plugin **does not** include this feature in this phase (IntelliJ's branch panel needs separate design).
  - This is explicitly listed as "intentionally unchanged: intellij/" in the PR description (per [[feedback_pr_summary_intentionally_unchanged]]).
- Do not touch the three `parseJolliApiKey / assertJolliOriginAllowed` mirrors.

## 12. Release and Rollback

- This is a new feature, does not affect existing data.
- Does not depend on Jolli API key / LLM quota — section works normally in any credential state.
- Rollback method: directly revert the PR introducing this section. No need to clean up any disk cache files (this design writes no persistent files).

## 14. Known Unresolved (Non-Blocking, Left for Follow-Up)

- Filter sessions by branch: requires extending session metadata to write branch tag. Not in this phase.
- IntelliJ port: not in this phase.
- Active session deletion / muting: not in this phase (not a core action of the active list).
- User-defined title override (local override): not in this phase. If requirement emerges, stack a user-edited title cache layer at the top of §5.1 priority.
- If Codex / Gemini / Copilot Chat add native title fields in the future: update §5.2 table + add entry to `SessionTitleResolver`, no impact on aggregator / protocol / frontend (this is the benefit of putting each source's title resolution in a separate resolver module).
- Possibility of "section-level refresh / independent section refresh": if future testing finds global refresh once scanning 7 sources + reading 4 sqlite + streaming N transcripts feels slow, then consider section-level refresh takeover. MVP stage global refresh is acceptable.
